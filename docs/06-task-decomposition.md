# 06 — Декомпозиция задач

## Принципы декомпозиции

- Каждая подзадача — одно изменение в одном файле (по возможности)
- Подзадача должна быть проверяема — есть конкретный признак завершения
- Не смешивать рефакторинг и новый функционал в одной подзадаче
- После каждой подзадачи — проверить что приложение компилируется: `dotnet build`
- Минимальное изменение: не трогать код за пределами задачи

## Шаблон декомпозиции

```
Задача: [описание]

Подзадачи:
1. [Прочитать] Изучить <файл.cs>, понять текущую реализацию
2. [Реализовать] Изменить <что именно> в <файл.cs>
3. [Проверить] dotnet build — 0 ошибок
4. [Тест] Проверить <конкретное поведение> (пользователь)
5. [Коммит] (только по запросу пользователя)
```

## Примеры декомпозиции

### Добавить новую модель whisper

```
1. AppSettings.cs — добавить новое значение в список WhisperModelOptions
2. MainWindow.xaml — убедиться что ComboBox подхватит новое значение (если список в XAML)
3. PathHelper.cs — проверить формат имени файла ggml-{model}.bin
4. dotnet build
5. download-model.ps1 -ModelName "new-model" — скачать модель
6. Тест: выбрать модель в настройках, запустить транскрипцию
```

### Исправить баг в постобработке

```
1. Воспроизвести баг — зафиксировать шаги
2. Прочитать FFmpegService.cs / WhisperService.cs / PipelineService.cs
3. Найти место ошибки (exit code, stdout, stderr)
4. Воспроизвести вручную в PowerShell (ffmpeg/whisper команду)
5. Исправить в коде
6. dotnet build
7. Тест: запустить постобработку с тестовым видео
8. Коммит: fix: <описание>
```

### Добавить новое поле в настройки

```
1. AppSettings.cs — добавить свойство с дефолтным значением
2. MainWindow.xaml — добавить UI-элемент в нужную карточку настроек
3. MainViewModel.cs — добавить [ObservableProperty] и при необходимости OnPropertyChanged
4. lang/ru/lang.json и lang/en/lang.json — добавить строки локализации
5. dotnet build
6. Тест: изменить настройку, сохранить, перезапустить, проверить что сохранилась
```

### Добавить новый тип маркера

```
1. Models/MarkerType.cs — добавить значение в enum
2. Converters/MarkerTypeToBrushConverter.cs — добавить цвет для нового типа
3. MainWindow.xaml — добавить кнопку в список маркеров (MarkerButtonStyle)
4. HotkeyService.cs — добавить поддержку хоткея для нового типа (если нужен)
5. HotkeyAction.cs — добавить значение в enum
6. AppSettings.cs — добавить биндинг по умолчанию в GetDefaultHotkeys()
7. lang/ru/lang.json и lang/en/lang.json — добавить loc_HkMarkerXxx
8. MainViewModel.cs — добавить команду AddXxxMarker
9. dotnet build
10. Тест: добавить маркер нового типа, проверить цвет, хоткей, Excel
```

### Обновить установщик (ClipNotes.Setup)

```
1. Прочитать ClipNotes.Setup/Pages/<страница>.xaml.cs — понять текущую логику
2. Внести изменения
3. dotnet build (проект ClipNotes.Setup)
4. .\rebuild-installers.ps1 — пересобрать
5. Тест: запустить Setup\ClipNotes-Setup.exe, пройти установку
```

### Добавить горячую клавишу

```
1. Models/HotkeyAction.cs — добавить значение в enum
2. AppSettings.cs → GetDefaultHotkeys() — добавить биндинг по умолчанию
3. MainViewModel.cs — добавить обработку в OnHotkeyPressed
4. lang/ru/lang.json и lang/en/lang.json — добавить loc_HkXxx
5. MainWindow.xaml — добавить строку в секцию Горячие клавиши
6. dotnet build
7. Тест: назначить хоткей, проверить что срабатывает
```

## Правило минимального изменения

- Не рефакторить код, который не связан с задачей
- Не добавлять комментарии к коду, который не менялся
- Не менять форматирование чужого кода
- Не добавлять error handling для сценариев, которые не могут возникнуть в этой задаче

---

## Декомпозиция сложных багов

Если непонятно где баг:

```
1. Добавить System.Diagnostics.Debug.WriteLine в подозрительные места
2. Запустить приложение, воспроизвести баг
3. Посмотреть app\Logs\YYYY-MM-DD.log
4. Сузить до конкретного метода
5. Воспроизвести вручную (PowerShell для FFmpeg/Whisper)
6. Исправить
7. Убрать временное логирование
```

---

## Работа с незнакомым кодом

При первом знакомстве с файлом:
1. Прочитать файл целиком (Read tool)
2. Определить: что делает, какие зависимости, какой публичный API
3. Не начинать менять до понимания всего файла
4. Если файл большой (>300 строк) — читать по секциям, начиная с публичных методов
