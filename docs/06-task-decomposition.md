# 06 — Декомпозиция задач

## Принципы декомпозиции

- Каждая подзадача — одно изменение в одном файле (по возможности)
- Подзадача должна быть проверяема — есть конкретный признак завершения
- Не смешивать рефакторинг и новый функционал в одной подзадаче
- После каждой подзадачи — проверить что приложение компилируется

## Шаблон декомпозиции

```
Задача: [описание]

Подзадачи:
1. [Прочитать] Изучить <файл.cs>, понять текущую реализацию
2. [Реализовать] Изменить <что именно> в <файл.cs>
3. [Проверить] Скомпилировать: dotnet build
4. [Тест] Проверить <конкретное поведение>
5. [Коммит] (только по запросу пользователя)
```

## Примеры декомпозиции реальных задач

### Задача: Добавить новую модель whisper

```
1. Прочитать AppSettings.cs — найти список поддерживаемых моделей
2. Добавить новое значение в enum/список моделей
3. Прочитать SettingsWindow.xaml — найти ComboBox выбора модели
4. Убедиться что новая модель появилась в UI (или добавить)
5. Проверить PathHelper.cs — формат имени файла модели корректен
6. Проверить: dotnet build
7. Тест: скачать модель через download-model.ps1 -ModelName "new-model"
8. Тест: запустить приложение, выбрать модель, протестировать транскрипцию
```

### Задача: Исправить баг в постобработке

```
1. Воспроизвести баг — зафиксировать точные шаги
2. Прочитать AudioProcessingService.cs
3. Найти место ошибки (exit code, stdout, stderr)
4. Воспроизвести вручную в PowerShell (ffmpeg/whisper команду)
5. Исправить в коде
6. dotnet build
7. Тест: запустить постобработку, проверить результат
8. Коммит: fix: <описание>
```

### Задача: Добавить новое поле в настройки

```
1. Прочитать AppSettings.cs — добавить свойство
2. Прочитать SettingsService.cs — убедиться что JSON сериализация подхватит
3. Прочитать SettingsWindow.xaml + .cs — добавить UI-элемент и биндинг
4. Прочитать MainViewModel.cs — использовать новое поле
5. dotnet build
6. Тест: изменить настройку, сохранить, перезапустить, проверить что сохранилось
```

### Задача: Рефакторинг сервиса

```
ВАЖНО: сначала убедиться что тесты/функционал работает ДО рефакторинга.

1. Прочитать сервис полностью
2. Выделить одну конкретную проблему (не всё сразу)
3. Изменить только эту проблему
4. dotnet build
5. Тест: прогнать весь чек-лист из 03-testing.md для этого сервиса
6. Коммит: refactor: <что именно>
```

### Задача: Новая фича (от пользователя)

```
1. Уточнить требования если неясно
2. Найти точку входа — какой файл нужно менять (Model? Service? ViewModel? View?)
3. Декомпозировать по слоям MVVM снизу вверх:
   a. Model — данные
   b. Service — логика
   c. ViewModel — команды и свойства
   d. View (XAML) — UI
4. Реализовывать по одному слою за подзадачу
5. После каждого слоя — dotnet build
6. Финальный тест — полный сценарий использования фичи
```

## Правило минимального изменения

- Не рефакторить код, который не связан с задачей
- Не добавлять комментарии к коду, который не менялся
- Не менять форматирование чужого кода
- Не добавлять error handling для сценариев, которые не могут возникнуть

## Декомпозиция сложных багов

Если непонятно где баг:

```
1. Добавить System.Diagnostics.Debug.WriteLine в подозрительные места
2. Запустить приложение, воспроизвести баг
3. Посмотреть Output в VS или добавить логирование в файл
4. Сузить до конкретного метода
5. Проверить аргументы/возвращаемые значения
6. Исправить
7. Убрать временное логирование
```

## Работа с незнакомым кодом

При первом знакомстве с файлом:
1. Прочитать файл целиком (Read tool)
2. Определить: что это делает, какие зависимости, какой публичный API
3. Не начинать менять до полного понимания
4. Если файл большой (>300 строк) — читать по секциям, начиная с публичных методов
